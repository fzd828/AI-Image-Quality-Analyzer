# 算法说明

本文说明 App 当前使用的画质评分指标、算法来源、参数含义和局限。所有指标都是本地传统计算机视觉规则，不是深度学习识别结果。

## 算法来源口径

本文里的“算法来源”指指标思想来源和工程取舍来源，不表示 App 接入了外部模型、云 API 或训练数据。当前评分来自传统图像处理 / 无参考图像质量评价中常见的手工特征：边缘焦点评价、亮度直方图、RMS/标准差对比度、RGB 通道平衡估计，再结合本项目样本验证后的阈值和权重。参数不是行业通用结论，而是为了本次 Android Demo 在速度、解释性和验证成本之间做出的工程设定。

## 清晰度 / 模糊

最终方法：在降采样后的灰度分析图上组合 3 个边缘信号：全局 Laplacian 方差、局部块 focused Laplacian 方差、Tenengrad 边缘能量。

算法来源：

- Laplacian 方差：来源于传统焦点/模糊检测思路，用二阶边缘响应的方差估计图像高频细节。边缘变化越多，方差通常越高。
- 局部块 focused Laplacian：来源于本项目验证后的工程改造。它沿用 Laplacian 方差，但只取局部最清楚的一部分区域，减少大面积平滑背景对主体清晰度的拖累。
- Tenengrad：来源于传统基于 Sobel 梯度能量的焦点评价。它用一阶边缘强度补充 Laplacian，但容易被噪点和密集纹理抬高。

清晰度校准升级：

- 保留全局 Laplacian 方差，用作整张图的整体边缘/细节响应。
- 增加 `focusedLaplacianVariance`：把分析图分成 `8 x 8` 网格，计算每个小块的 Laplacian 方差，再取最清晰的前 25% 小块求平均。这样能更关注主体或局部细节，减少平滑背景、浅景深背景对清晰度的拖累。
- 增加 `tenengrad`：使用 Sobel 梯度能量作为辅助边缘证据。噪点和复杂纹理也会抬高它，所以只作为低权重辅助信号。

原始信号含义：

- 全局 Laplacian 高：整张图有更多边缘和细节响应。
- 局部 focused Laplacian 高：至少部分区域存在较清楚的细节。
- Tenengrad 高：Sobel 边缘能量较强。
- 三者都低：更可能是失焦、运动模糊或低细节画面。

参数：

- 分析图尺寸：由 App 内分析模式决定，本轮最终验证统一使用 `精细模式 2560`。
- Laplacian 核：用于检测二阶边缘变化。
- 局部网格：`8 x 8`。
- 局部块选择：取方差最高的前 25% 小块。
- 清晰度权重：focused Laplacian 60%，全局 Laplacian 25%，Tenengrad 15%。
- 归一化半分值：focused Laplacian 150，全局 Laplacian 400，Tenengrad 4000。

升级原因：

第一版只使用全局 Laplacian 方差，简单、快速、可解释，但验证时发现两类问题：自然照片或浅景深照片会被大面积平滑背景拉低；噪点、复杂纹理和高频背景细节又可能把边缘响应抬高。升级后用局部块 Laplacian 关注最清楚区域，同时保留全局稳定性，Tenengrad 只做辅助。

局限：

- 噪点和压缩伪影可能被误认为清晰细节。
- 复杂纹理、高 ISO 噪点仍会抬高 Laplacian 和 Tenengrad。
- 平滑但实际对焦正确的场景可能得分偏低。
- 艺术模糊、运动模糊、浅景深需要人工解释，因为 App 没有主体识别或拍摄意图理解。
- 如果分析图尺寸变化，阈值和分数也会变化。

## 曝光

最终方法：使用亮度直方图和明暗裁剪比例。

算法来源：

曝光项来源于传统亮度直方图分析：用平均亮度判断整体偏亮/偏暗，用暗部和亮部裁剪比例估计欠曝或过曝风险。这是无参考图像质量评价里的常见工程规则，不依赖场景识别。

原始信号：

- 平均亮度。
- 欠曝像素比例。
- 过曝像素比例。

参数：

- 亮度计算：`Y = 0.299R + 0.587G + 0.114B`。
- 理想平均亮度：`128`，表示 8-bit 图像的中间调附近。
- 欠曝阈值：亮度 `<= 20` 的像素计入暗部裁剪。
- 过曝阈值：亮度 `>= 235` 的像素计入亮部裁剪。
- 平均亮度惩罚权重：`35`。
- 裁剪比例惩罚权重：`127`，让明显欠曝/过曝比轻微亮度偏移扣分更重。

局限：

- 夜景可能是有意偏暗。
- 雪景、天空、高调照片可能是有意偏亮。
- 全局直方图无法判断主体局部曝光是否正确。

## 对比度

最终方法：使用灰度对比统计，主要是标准差 / RMS 风格的对比度。

算法来源：

对比度项来源于 RMS/标准差对比度思路：把灰度值围绕平均亮度的离散程度作为画面层次强弱的近似指标。标准差越高，通常表示明暗变化越明显。

参数：

- 统计对象：降采样分析图的灰度亮度数组。
- 目标标准差：`55`。达到或超过该值时，对比度子分接近满分。
- 分数归一化：`contrastStdDev / 55 * 100`，再限制在 `0-100`。

局限：

- 低对比可能是艺术风格。
- 高对比不一定更好，也可能隐藏暗部或高光细节。

## 偏色 / 色彩平衡

最终方法：排除极暗和极亮像素后，估计 RGB 通道不平衡程度。

算法来源：

偏色项来源于通道平衡 / 灰世界假设的简化工程规则：在中间亮度区域，如果 RGB 三个通道平均值偏离共同中性值较多，就提示白平衡或色光风险。它不是专业色彩管理，也不理解拍摄意图。

参数：

- 参与统计的像素亮度范围：`20 < Y < 235`，避免纯黑、纯白和裁剪区域影响通道均值。
- 中性值：`(avgR + avgG + avgB) / 3`。
- 偏色距离：`sqrt((avgR-neutral)^2 + (avgG-neutral)^2 + (avgB-neutral)^2)`。
- 可接受偏色距离上限：`80`，超过后偏色子分会明显下降。

局限：

- 暖光、日落、霓虹、滤镜可能是有意色彩。
- 没有场景语义时，只能估计色彩偏移风险，不能等同于专业白平衡判断。

## 综合分

算法来源：

综合分不是训练出来的模型分数，而是本项目为作业 Demo 设计的启发式加权融合。权重来源于项目目标：优先体现清晰度和曝光，同时保留对比度与偏色风险，让结果能解释、能复现、能被日志和截图验证。

最终公式：

```text
overall = 0.35 * sharpness
        + 0.30 * exposure
        + 0.20 * contrast
        + 0.15 * color_or_noise
```

这个公式简单、可解释，适合本次作业 Demo。但它也有局限：某一项严重问题可能被其他高分项稀释。`overexposed_sky_02.jpg` 就是最终反例：曝光有明显问题，但清晰度、对比度、偏色得分较高，导致综合分仍偏高。

## 诊断说明层

算法来源：

诊断说明层来源于规则映射，不是运行时大模型生成。它把综合分、低分子项和少量风险条件转换成“综合判断 / 主要问题 / 建议”三段式文案，目的是降低用户和评审理解成本。

诊断说明 V2 只改变结果文案，不改变核心算法。它会按严重程度列出最多三个弱项，区分欠曝和过曝，并在“暗场景 + 很高边缘响应”时增加保守的噪声/纹理风险提示。

参数：

- 综合判断阈值：`>= 80` 且无明显弱项时为整体较好，`>= 60` 时为整体可用但存在风险，否则提示整体质量偏低。
- 弱项阈值：子分 `< 60` 会进入主要问题候选。
- 最多展示问题数：`3`。
- 噪声/纹理风险只在高边缘响应、偏暗/低曝光、高对比同时出现时提示。

注意：当前 App 只有噪声/纹理风险提示，没有独立噪声检测指标。因此 noisy 样本仍属于算法局限展示，不能声称已经具备专业噪声检测能力。

## 性能与大图

App 会记录每张图片的分析耗时，并记录：

- 原始图片尺寸。
- 分析图尺寸。
- 是否发生降采样。

这些信息用于证明单图分析耗时可接受，并证明大图不会直接按原始 12MP+ 尺寸硬解分析，从而降低 OOM 风险。

## HEIC 兼容性

HEIC 只作为兼容性说明，不作为最低交付格式。

原因：

- Android 对 HEIC 的支持依赖系统版本、设备编解码器和解码 API 行为。
- 作业要求至少 3 种常见格式，本项目核心承诺是 JPEG、PNG、WebP。
- 如果设备和 API 稳定支持，HEIC 可以作为后续加分验证，但不写入本轮完成判据。
