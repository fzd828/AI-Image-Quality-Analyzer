# AI 协作说明

## 基本原则

AI 在本项目中是研究、拆解、生成草稿、提出候选方案和整理证据的助手；人负责选择方向、复核结果、发现偏差、纠错取舍和最终验证。这里的“AI”不是 App 接入云端大模型，也不是使用深度学习模型识别图片质量，最终可信度来自可复现的代码、日志、截图和人工复核。

## 我如何用 AI 分发任务

原始作业要求同时包含 App 能力、验证证据、README、约束决策、AI 协作和可复现仓库。我使用 Superpowers + OpenSpec 把这些要求拆开：Superpowers 用来做头脑风暴、计划、工作流分发和过程复盘；OpenSpec 用来把需求固化成 proposal、design、specs、tasks，并形成 WS0-WS8 工作流：先跑通 Android 工程，再做选图预览、格式解码、大图安全、质量指标、评分说明、样本验证、证据整理和最终文档包。

人的判断是：不把项目做成“功能越多越好”，而是每轮只执行一个工作包，并让每个工作包留下证据。这样能把评审关心的链路固定下来：代码 -> APK -> 样本 -> 日志 -> 截图 -> 文档结论。

## 我如何用 AI 迭代项目

| 阶段 | AI 帮助做的事 | 人的判断和验证 |
|---|---|---|
| 需求定义 | 把作业图中的核心信号拆成目标、非目标、完成判据和交付物 | 选择选题 1，确认项目是 Android 本地图片质量分析工具 |
| 任务拆分 | 用 Superpowers 做工作流分发和复盘，用 OpenSpec 记录 proposal、design、specs、tasks，并提示每轮留下验证证据 | 决定按工作流推进，不临时扩展云 API、账号、相册管理等功能 |
| App 实现 | 辅助生成 Kotlin + Compose 代码、解码流程、评分逻辑和诊断文案 | 在模拟器和 Redmi K70E 上运行，检查选图、预览、分析和截图 |
| 算法选择 | 提出 Laplacian、Tenengrad、曝光直方图、对比度、偏色、噪点风险等候选指标 | 选择传统 CV 指标，放弃深度学习和云端识别，保证本地可跑、可解释、可复现 |
| 证据整理 | 建议用日志、截图、样本说明、反例文档形成证据链 | 核对最终日志、截图路径、设备来源和样本来源，避免把模拟器写成真机 |
| 文档收尾 | 帮助重组 README、约束决策、验证证据、反例、复现说明和 AI 协作说明 | 根据面试官阅读成本继续改成中文、补解释、修正 Match/Partial 等不清楚的地方 |

## 我怎么发现需要优化

优化不是凭空加功能，而是从验证和复核中发现问题：

- 从截图和日志看出，大图需要安全降采样，否则会有 OOM 风险或分析耗时不可控。
- 从样本对比看出，单纯全局清晰度容易受噪点、纹理和夜景影响，所以需要局部块清晰度和 Tenengrad 辅助解释。
- 从反例看出，过曝火车天空虽然局部问题明显，但总分仍可能约 80/100；香港夜景噪点明显，但清晰度和对比度会偏高。这说明评分不是“真理”，必须保留人工解释和反例。
- 从文档复核看出，Match/Partial、日志/截图/数据/对比、外部样本/真机截图等概念如果不解释，面试官会看不懂。
- 从实际打开文件看出，CSV 用 Excel 双击会乱码，所以最终日志改成 UTF-8 with BOM，降低评审成本。

## 我怎么选择优化方案

选择优化方案时，我优先考虑四个标准：是否直接对应作业要求，是否能在 Android 本地稳定运行，是否能用日志和截图验证，是否能让面试官快速理解。基于这个标准，我没有把时间投入深度学习、RAW/HEIC 专业格式、完整相册管理或复杂图表，而是把精力放在已有项目的关键链路上：

- 优化预览体验，让选图后能立即看到图片和基础信息。
- 做大图安全降采样，降低内存风险。
- 增加极速/精细模式，在速度和指标细节之间做取舍。
- 优化局部块清晰度算法，减少全局指标被背景、噪点或局部纹理误导。
- 改写评价文案，让评分结果能说明“为什么”，也说明局限。
- 保留反例验证，主动展示算法会失准的场景，而不是只放好看的成功样本。

## 我怎么验证优化有效

验证分成四类证据：

- 日志：`logs/final_validation_log.csv` 记录最终样本、设备、格式、分数、匹配状态和截图路径。
- 截图：`screenshots/README.md` 和 `screenshots/` 区分 Android Emulator、Redmi K70E、外部样本截图和手机截图生成副本。
- 数据：`deliverables/docs/validation-evidence.md` 汇总 JPEG/PNG/WebP、大图、清晰/模糊/过曝/欠曝/噪点等样本表现。
- 对比：`deliverables/docs/counterexamples.md` 保留评分明显不完美的反例，用来说明算法边界。

这些证据经过人工复核：App 真机运行过，截图和日志能互相对应，分数与人工判断做过 Match/Partial 对照，反例解释不是 AI 直接下结论，而是结合实际样本表现后保留的判断。

## AI 生成、人修改/验证的边界

| 内容 | AI 的作用 | 人的作用 |
|---|---|---|
| 需求和计划 | 辅助拆解作业要求，用 Superpowers 组织工作流，用 OpenSpec 固化需求和任务 | 决定选题、边界、优先级和每轮是否继续 |
| 代码和算法 | 生成候选实现，提出传统 CV 指标组合 | 运行 App，观察结果，确认是否保留或调整 |
| 诊断文案 | 草拟清晰度、曝光、噪点、对比度、偏色等解释 | 改写成不过度承诺的中文说明 |
| 测试建议 | 提醒覆盖格式、大图、真机、反例和文档证据 | 实际执行模拟器/真机验证，保存截图和日志 |
| 文档整理 | 整理 README、验证证据、反例、复现和协作说明 | 核对事实边界，修正旧日志、旧描述、英文说明和不清楚的术语 |

## 关键修正案例

| AI 输出或早期倾向 | 问题 | 最终修正 |
|---|---|---|
| 做深度学习或云端 AI 更像“AI 项目” | 部署、隐私、网络、模型选择和解释成本过高 | 改成本地传统 CV，并说明 AI 只参与构建和验证 |
| 支持 RAW、完整 HEIC 或更多专业格式 | 编解码、色彩空间、样本来源和兼容性成本高 | 核心承诺限定为 JPEG/PNG/WebP，HEIC 只做边界说明 |
| 做完整相册管理 | 会增加批量导入、缩略图缓存、权限和状态维护 | 只做图片选择、预览和单图分析，贴合作业核心 |
| 只用全局 Laplacian 判断清晰度 | 容易被背景、纹理、噪点和夜景误导 | 增加局部块清晰度和 Tenengrad 辅助，并保留反例 |
| 只输出原始指标或单一综合分 | 面试官看不出 App 为什么这样评价，也难以判断结果是否可信 | 改成综合分 + 清晰度/曝光/对比度/偏色等子分，并配套可读诊断说明 |
| 把 App 评分当成绝对正确 | 实际样本会出现 Partial：分数大致可用，但局部问题或人工判断需要补充解释 | 在日志和文档中加入 Match/Partial 对照，用人工判断说明评分边界 |
| 只说“有验证证据” | 面试官仍要自己找日志、截图、数据和对比 | 把四类证据写清楚，并链接到对应文件 |
| 文档中出现英文或历史日志 | 会增加中文面试官阅读成本，或误解最终证据 | 改成中文说明，最终证据统一指向 `logs/final_validation_log.csv` |

## 结论

这个项目不是 AI 全自动完成。AI 帮我做任务分发、迭代建议、文档重组和优化方案候选；人负责选择题目、控制范围、运行设备、检查截图、复核分数、识别反例、修正文档和最终判断。最终提交体现的是人机协作透明度：AI 提高效率，人负责判断质量。
